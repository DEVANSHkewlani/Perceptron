01 Architecture Overview

How DCA Wraps ShopCore

ShopCore is a 4-service e-commerce backend: an API Gateway on port 8000, a Product Service on 8001, an Order Service on 8002, and a User Service on 8003, backed by PostgreSQL, Redis, and Redpanda (Kafka-compatible). The DCA sits entirely outside these services — it observes them through instrumentation hooks and acts on them through controlled API calls and Docker operations. ShopCore never imports any DCA code.

03 Adapter Build Guide

Building the New Adapters

Four adapters already exist from Phases 1–11 (log, api, database, queue). You need to build three new ones for the ShopCore chaos lab and extend one existing one.

New Adapter 1: RedisMonitor

Polls INFO all from Redis every 15 seconds and emits events when memory, evictions, or hit ratio cross thresholds.

instrumentation/adapters/redis_monitor.py
import asyncio, logging, time
from datetime import datetime, timezone
import redis.asyncio as aioredis
from cognitive_perception.schema.event import CognitiveEvent, Severity, SourceType

class RedisMonitor:
    def __init__(self, config: dict, producer):
        self.url = config["url"]
        self.config = config
        self.producer = producer
        self._prev_evictions = 0
        self._prev_hits = self._prev_misses = 0

    async def run(self):
        r = await aioredis.from_url(self.url)
        while True:
            try:
                info = await r.info("all")
                await self._check(info)
            except Exception as e:
                # Redis itself is unreachable — emit a service_unreachable event
                await self._emit("service_unreachable", Severity.CRITICAL,
                    {"error": str(e)}, ["cache:shopcore-redis"])
            await asyncio.sleep(self.config.get("poll_interval_s", 15))

    async def _check(self, info):
        # 1. Memory pressure check
        maxmem = info.get("maxmemory") or info["total_system_memory"]
        used = info["used_memory"]
        pct = used / maxmem if maxmem else 0
        if pct >= self.config.get("memory_pct_critical", 0.90):
            await self._emit("cache_eviction_spike", Severity.CRITICAL, {
                "used_bytes": used, "max_bytes": maxmem,
                "used_pct": round(pct * 100, 1),
            }, ["cache:shopcore-redis"])

        # 2. Eviction spike check
        curr_evictions = info.get("evicted_keys", 0)
        delta = curr_evictions - self._prev_evictions
        warn_delta = self.config.get("eviction_warn_delta", 10)
        if delta >= warn_delta:
            await self._emit("cache_eviction_spike", Severity.HIGH, {
                "evictions_this_interval": delta,
                "total_evictions": curr_evictions,
            }, ["cache:shopcore-redis"])
        self._prev_evictions = curr_evictions

        # 3. Hit ratio check
        hits = info.get("keyspace_hits", 0)
        misses = info.get("keyspace_misses", 0)
        delta_hits = hits - self._prev_hits
        delta_misses = misses - self._prev_misses
        total = delta_hits + delta_misses
        if total > 0:
            ratio = delta_hits / total
            if ratio < self.config.get("hit_ratio_warn", 0.70):
                await self._emit("cache_hit_ratio_dropped", Severity.HIGH, {
                    "hit_ratio": round(ratio, 3),
                    "hits": delta_hits, "misses": delta_misses,
                }, ["cache:shopcore-redis"])
        self._prev_hits, self._prev_misses = hits, misses

    async def _emit(self, event_type, severity, payload, entity_refs):
        event = CognitiveEvent(
            timestamp=datetime.now(timezone.utc),
            source_type=SourceType.METRIC,
            source_id="cache:shopcore-redis",
            event_type=event_type,
            severity=severity,
            payload=payload,
            entity_refs=entity_refs,
            confidence=0.97,
            tags=["shopcore", "cache", "redis"],
        )
        await self.producer.send_and_wait(
            "cognitive.events", event.model_dump_json().encode()
        )

New Adapter 2: FileWatchdog

Uses the watchdog library on Linux (backed by inotify) to detect any write to nginx config files within seconds.

instrumentation/adapters/file_watchdog.py
import asyncio, hashlib, logging
from pathlib import Path
from watchdog.observers import Observer
from watchdog.events import FileSystemEventHandler, FileModifiedEvent
from cognitive_perception.schema.event import CognitiveEvent, Severity, SourceType
from datetime import datetime, timezone

class ConfigFileHandler(FileSystemEventHandler):
    def __init__(self, source_id, producer, loop):
        self.source_id = source_id
        self.producer = producer
        self.loop = loop
        self._hashes: dict[str, str] = {}

    def on_modified(self, ev):
        if ev.is_directory: return
        path = Path(ev.src_path)
        try:
            content = path.read_bytes()
            new_hash = hashlib.sha256(content).hexdigest()
            old_hash = self._hashes.get(str(path), "")
            if new_hash == old_hash: return     # metadata touch, not real change
            self._hashes[str(path)] = new_hash
        except: return

        event = CognitiveEvent(
            timestamp=datetime.now(timezone.utc),
            source_type=SourceType.FILE,
            source_id=self.source_id,
            event_type="config_file_modified",
            severity=Severity.MEDIUM,
            payload={"file": str(path), "sha256": new_hash},
            entity_refs=[self.source_id, "infra:nginx"],
            confidence=0.99,
            tags=["shopcore", "config", "nginx"],
        )
        asyncio.run_coroutine_threadsafe(
            self.producer.send_and_wait("cognitive.events", event.model_dump_json().encode()),
            self.loop
        )

class FileWatchdog:
    def __init__(self, config, producer, loop):
        self.config = config
        self.producer = producer
        self.loop = loop

    async def run(self):
        handler = ConfigFileHandler(self.config["id"], self.producer, asyncio.get_event_loop())
        observer = Observer()
        for path in self.config.get("watch_paths", []):
            observer.schedule(handler, path, recursive=True)
        observer.start()
        try:
            while True: await asyncio.sleep(1)
        finally:
            observer.stop()
            observer.join()

New Adapter 3: BrowserSDK Receiver

A FastAPI endpoint that receives POST requests from the React frontend's cognitive_sdk.js and normalizes them into CognitiveEvent objects.

instrumentation/adapters/browser_sdk_endpoint.py — mount on your existing FastAPI app
from fastapi import APIRouter, Request
from cognitive_perception.schema.event import CognitiveEvent, Severity, SourceType
from datetime import datetime, timezone

browser_router = APIRouter(prefix="/perception")

VITAL_THRESHOLDS = {"lcp_ms": 2500, "fid_ms": 100, "cls": 0.10}

@browser_router.post("/browser-events")
async def ingest_browser_events(request: Request):
    producer = request.app.state.kafka_producer
    body = await request.json()

    for raw in (body if isinstance(body, list) else [body]):
        event_type, severity, payload = _normalize(raw)
        event = CognitiveEvent(
            timestamp=datetime.now(timezone.utc),
            source_type=SourceType.USER_EVENT,
            source_id="frontend:shopcore-spa",
            event_type=event_type,
            severity=severity,
            payload=payload,
            entity_refs=["frontend:shopcore-spa"],
            confidence=0.90,
            tags=["shopcore", "frontend", raw.get("page", "unknown")],
        )
        await producer.send_and_wait("cognitive.events", event.model_dump_json().encode())
    return {"status": "accepted"}

def _normalize(raw):
    kind = raw.get("type", "")
    if kind == "web_vital":
        name = raw.get("name"); val = raw.get("value", 0)
        threshold = VITAL_THRESHOLDS.get(f"{name}_ms", VITAL_THRESHOLDS.get(name, 99999))
        sev = Severity.HIGH if val > threshold * 2 else Severity.MEDIUM
        return "web_vital_degraded", sev, raw
    elif kind == "rage_click":
        return "user_frustration_detected", Severity.MEDIUM, raw
    elif kind == "checkout_error":
        return "checkout_failure", Severity.HIGH, raw
    elif kind == "js_exception":
        return "frontend_exception", Severity.HIGH, raw
    return "browser_event", Severity.INFO, raw

Extended: DatabaseMonitor — add idle-in-transaction + lock detection

Your existing DatabaseMonitor needs two new queries added to its poll loop:

additions to existing DatabaseMonitor._poll()
# ADD to your DatabaseMonitor poll loop alongside slow query detection:

# ── Idle-in-transaction connections ──────────────────────────────
idle_result = await conn.fetch("""
    SELECT pid, usename, application_name,
           EXTRACT(EPOCH FROM now() - query_start) AS idle_s
    FROM pg_stat_activity
    WHERE state = 'idle in transaction'
      AND pid != pg_backend_pid()
""")
for row in idle_result:
    if row["idle_s"] > config.get("idle_in_transaction_warn_s", 60):
        await emit("connection_leak_detected", Severity.MEDIUM, {
            "pid": row["pid"], "idle_s": row["idle_s"],
            "application": row["application_name"],
        })

# ── Lock waits ────────────────────────────────────────────────────
lock_result = await conn.fetch("""
    SELECT blocked.pid AS blocked_pid,
           blocking.pid AS blocking_pid,
           EXTRACT(EPOCH FROM now() - blocked_activity.query_start) AS wait_s,
           blocked_activity.query AS blocked_query
    FROM pg_locks blocked
    JOIN pg_locks blocking ON blocking.transactionid = blocked.transactionid
      AND blocking.pid != blocked.pid
    JOIN pg_stat_activity blocked_activity ON blocked.pid = blocked_activity.pid
    WHERE NOT blocked.granted
""")
for row in lock_result:
    if row["wait_s"] > config.get("lock_wait_warn_s", 5):
        await emit("lock_wait_timeout", Severity.MEDIUM, {
            "blocked_pid": row["blocked_pid"],
            "blocking_pid": row["blocking_pid"],
            "wait_s": row["wait_s"],
            "query_preview": row["blocked_query"][:200],
        })

06 Multi-Step Execution Flows

How the Agent Executes Complex Remediations

Fixing a real anomaly is never one action. It's a sequence with verification steps, fallbacks, and feedback at each stage. Below are the exact multi-step plans for the most complex scenarios.

Flow A: DB Pool Exhaustion → Full Recovery

TRIGGER: event_type=connection_pool_exhausted severity=CRITICAL entity_refs=[svc:product-service, db:shopcore-postgres] PLAN: Step 1 — TERMINATE_IDLE_CONNECTIONS (DatabaseHandler) ├─ Execute: pg_terminate_backend() for all idle-in-transaction > 30s ├─ Verify: SELECT count(*) FROM pg_stat_activity WHERE state='idle in transaction' │ → expect count < 5 within 10s ├─ Timeout: 15s └─ on_failure → escalate to Step 3 immediately Step 2 — VERIFY_POOL_RECOVERY (observation, no action) ├─ Wait up to 30s for pool_usage to drop below 70% in world model ├─ Success condition: db:shopcore-postgres.pool_used_pct < 0.70 └─ on_failure → proceed to Step 3 Step 3 — RESTART_CONTAINER [product-service] (ServiceHandler) ├─ Execute: docker.containers.get("product-service").restart() ├─ Verify: GET http://localhost:8001/health → 200 OK within 45s ├─ Timeout: 60s └─ on_failure → send_alert + request_human_approval Step 4 — FEEDBACK_PUBLISH (AgentEventAdapter) └─ Publish action_completed event to cognitive.events with: - steps_executed, elapsed_total_s, final_pool_pct - outcome: success | partial | escalated EXPECTED TOTAL DURATION: 30–90s ROLLBACK: None needed — connection termination is safe, restart is idempotent

Flow B: Kafka Consumer Lag → Unblock + Drain

TRIGGER: event_type=consumer_lag_high payload.consumer_lag > 100 entity_refs=[queue:order-events, svc:order-service] PLAN: Step 1 — RESTART_KAFKA_CONSUMER (QueueHandler) ├─ Execute: docker.containers.get("order-service").restart() ├─ Verify: GET http://localhost:8002/health → 200 OK within 30s ├─ Timeout: 45s └─ on_failure → skip to Step 3 Step 2 — MONITOR_LAG_DRAIN (observation) ├─ Poll consumer lag every 15s for up to 5 minutes ├─ Success condition: lag < 10 (fully drained) ├─ Acceptable: lag decreasing at rate > 5/poll └─ on_failure (lag not decreasing after 3 polls) → Step 3 Step 3 — SCALE_ORDER_SERVICE (ServiceHandler) [if lag still > 500] ├─ Execute: docker-compose scale order-service=2 ├─ Verify: two order-service containers healthy ├─ Timeout: 60s └─ Schedule scale_down after 15 minutes or lag=0 Step 4 — RESET_CONSUMER_OFFSET [requires_approval] (QueueHandler) ├─ Only reached if lag > 1000 and not decreasing after 10min ├─ POST approval request to Slack with: current lag, messages that will be skipped ├─ On approval: kafka-python AdminClient.alter_consumer_group_offsets(to=latest) └─ on_no_approval after 15min → escalate_to_human_incident EXPECTED TOTAL DURATION: 2–15 minutes depending on lag depth

Flow C: Config File Tamper → Validate and Heal

09 Detection Pipeline

From Raw Signal to CognitiveEvent

Detection is the first pillar. The full pipeline from raw signal to a routed event in memory takes under 2 seconds for push-based sources and under 15 seconds for polled sources. Here is every step.

Step 1 · Raw Signal
A container emits a log line, Prometheus fires an alert, a health poll times out, a Redis INFO value crosses a threshold, a file changes on disk, or the frontend SDK POSTs a web vital.
Step 2 · Adapter Ingestion
The appropriate adapter receives the raw signal. Fluent Bit tails container logs and writes to the raw.logs Kafka topic. Pollers run on asyncio task timers. File watchdog uses inotify callbacks.
Step 3 · Normalizer
Each adapter has a normalizer function: raw input → validated CognitiveEvent. Extracts event_type, entity_refs, severity, and assigns a confidence score (0.55–0.99 depending on signal quality).
Step 4 · Schema Validation
Pydantic validates the CognitiveEvent. Invalid events go to cognitive.perception_failures for retry. Valid events go to cognitive.events. Zero silent drops — every failure is itself an event.
Step 5 · Memory Router
Reads from cognitive.events. Routes: all events → episodic (TimescaleDB); entity_refs → Neo4j graph update; severity HIGH+ → working memory (Redis); known trigger patterns → procedural memory activation.
Step 6 · World Model Update
The world model's background loop reads new events from the bus and updates entity states, temporal profiles, causal chains, and anomaly registry. The situation assessor re-ranks active anomalies.
Step 7 · Monitor Agent Trigger
The monitor agent subscribes to working memory updates. When severity CRITICAL or HIGH events land, it triggers the reasoning engine with a fresh situation summary. Medium events are batched every 60s.
Step 8 · Fast-Path Check
Before calling the LLM, the reasoning engine checks the rule engine: does this event_type exactly match a known playbook with confidence ≥ 0.80? If yes, skip LLM and go straight to planning. Typically covers 60–70% of scenarios after 3+ successful runs.

11 Remediation Execution

The Executor in Detail

The executor receives a Plan from the planning system — a structured sequence of steps with preconditions, timeouts, and on_failure branches. It executes each step, verifies its outcome against the world model, and advances or rolls back.

PlanRunner Implementation

remediation/plan_runner.py
import asyncio, logging, time
from datetime import datetime, timezone
from .action_registry import ActionRegistry

class PlanRunner:
    def __init__(self, registry: ActionRegistry, world_model, producer):
        self.registry = registry
        self.world_model = world_model
        self.producer = producer

    async def execute(self, plan: dict, trigger_event) -> dict:
        context = {"plan_id": plan["id"], "results": [], "start_time": time.monotonic()}
        for step in plan["steps"]:
            # Check preconditions
            if step.get("precondition") and not await self._check(step["precondition"]):
                context["results"].append({"step": step["id"], "skipped": True, "reason": "precondition_not_met"})
                continue

            # Handle approval gates
            if self.registry.requires_approval(step["action"]):
                approved = await self._request_approval(step, trigger_event)
                if not approved:
                    context["results"].append({"step": step["id"], "skipped": True, "reason": "approval_denied"})
                    continue

            # Execute the action
            handler, params = self.registry.get_handler_for_event(step["action"]) or (None, {})
            if not handler:
                continue
            t0 = time.monotonic()
            try:
                result = await asyncio.wait_for(
                    handler.execute(step["action"], {**params, **step.get("params", {}), "event": trigger_event}),
                    timeout=step.get("timeout_s", 60)
                )
            except asyncio.TimeoutError:
                result = {"success": False, "reason": "timeout"}
            except Exception as e:
                result = {"success": False, "reason": str(e)}

            elapsed = time.monotonic() - t0
            context["results"].append({"step": step["id"], "result": result, "elapsed_s": round(elapsed, 2)})

            # Verify success condition
            if step.get("success_condition") and not await self._verify(step["success_condition"]):
                if step.get("on_failure") == "escalate":
                    context["escalated"] = True
                    break
                elif step.get("on_failure") == "rollback":
                    await self._rollback(plan, context)
                    break

        context["total_elapsed_s"] = round(time.monotonic() - context["start_time"], 2)
        await self._publish_feedback(plan, trigger_event, context)
        return context

    async def _verify(self, condition: dict) -> bool:
        # Query world model for current state, check condition
        entity_state = await self.world_model.get_entity_state(condition["entity"])
        metric = entity_state.get(condition["metric"])
        op = condition["op"]  # lt, gt, eq
        threshold = condition["value"]
        if op == "lt": return metric is not None and metric < threshold
        if op == "gt": return metric is not None and metric > threshold
        if op == "eq": return metric == threshold
        return False

    async def _request_approval(self, step, trigger_event) -> bool:
        # Post to Slack and wait on approval_events queue (timeout 15 min)
        import httpx
        payload = {
            "text": f"*DCA Approval Required*\nAction: `{step['action']}`\nTrigger: `{trigger_event.event_type}` on `{trigger_event.source_id}`\nReply `/approve {step['id']}` or `/deny {step['id']}`"
        }
        async with httpx.AsyncClient() as c:
            await c.post(SLACK_WEBHOOK, json=payload)
        # Block and wait for callback — see /approvals/respond endpoint
        try:
            decision = await asyncio.wait_for(
                self._approval_queue.get(),
                timeout=900  # 15 minute window
            )
            return decision == "approved"
        except asyncio.TimeoutError:
            return False
12 Feedback & Learning

Closing the Loop

The feedback loop is what turns the executor into a learning system. After every plan execution, the system records the outcome, updates procedural memory's confidence scores, and reinforces causal relationships in the knowledge graph.

The 7-Step Feedback Cycle

  1. Action completes — PlanRunner calls _publish_feedback() with full plan context, elapsed time, and per-step results. This posts an action_completed (or action_failed) event to cognitive.events.
  2. Perception picks it up — The agent_event normalizer processes the feedback event like any other CognitiveEvent. It gets timestamped, entity-referenced, and validated. It flows through the memory router.
  3. Episodic memory records the full context — TimescaleDB stores: the original trigger event, the situation summary that was generated, the decision that was made, every plan step and its outcome, and the total elapsed time.
  4. World model updates — The entity that was remediated has its state refreshed. The anomaly is removed from the active anomaly registry. Causal chain edges are updated: the RESOLVES relationship between the action and the anomaly is strengthened.
  5. Procedural memory updates confidence — The playbook entry for this event_type → action mapping gets its success count incremented. After 3 successes: fast-path eligible. After 10+ with <20% failure rate: fully trusted. Failures increment failure count and reduce confidence.
  6. Knowledge graph learns relationships — If action A consistently resolves event type B on entity C, a RESOLVES edge is added or strengthened between the action node and the event type node. If two event types consistently co-occur (e.g. slow queries always precede pool exhaustion), a HISTORICALLY_CORRELATED edge is added.
  7. Situation assessor recalibrates — The next time a similar situation arises, the assessor will rank it differently — higher if previously hard to resolve, lower if it consistently resolves quickly. The LLM's context is continuously improving.
