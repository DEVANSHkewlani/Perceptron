01 / Overview
What This Dashboard Is

This dashboard is the complete observability surface for your cognitive architecture — a single-pane-of-glass view across all 11 phases you have built. It is not a metrics collector. It is a live cognitive mirror: you watch the architecture perceive, remember, reason, plan, and execute in real time. When you connect the Chaos Lab (an external service or website under test), every signal it emits becomes visible here — from the raw Kafka event through memory storage, world model update, reasoning trace, agent action, and feedback closure.

⚡
Real-Time Event Stream
Live feed of every normalized event flowing through Kafka. Filterable by source, severity, event_type, and entity_ref.
🧠
Four-Layer Memory View
Working, episodic, semantic (graph), and procedural memory — current state, utilization, recent reads/writes, TTL expiry countdowns.
🌐
Live World Model
The architecture's current belief about the environment: entity states, causal chains, anomaly registry, predictive state, and uncertainty map.
🤖
Agent Fleet Console
Every active agent — monitor, planner, executor, memory, prediction, specialist — with current task, status, and last action trace.
🔥
Chaos Lab Panel
When connected to a target service/website, injects faults and visualizes exactly how the architecture detects, reasons, and responds.
🔁
Feedback Loop Tracker
Closed-loop trace from action execution → effect observation → memory update → world model delta → strategy reinforcement.
Design Principle
The dashboard is read-only during normal operation — it observes and displays. The only writes it performs are: injecting chaos events (Chaos Lab) and issuing human approval/rejection for plans that require it. Everything else is pure observation.

07 / Dashboard Tech Stack
Recommended Implementation Stack

The dashboard backend aggregates from all architecture components. The frontend renders in real time via WebSocket.

LayerTechnologyWhy
Frontend frameworkReact + TypeScriptComponent model matches panel architecture. TypeScript catches schema mismatches early.
Real-time transportWebSocket (native) or Socket.IODashboard needs sub-second updates for live panels. REST polling is too slow for event stream.
Dashboard stateZustand or JotaiLightweight atom-based state. Each panel subscribes to its slice — no unnecessary re-renders.
Graph visualizationSigma.js or React FlowSigma.js handles large graphs (1000+ nodes) in WebGL. React Flow is better for architecture flowchart (smaller, interactive).
Time-series chartsRecharts or uPlotuPlot is faster for dense data. Recharts is easier to style consistently.
Architecture diagramReact FlowNode-edge graph with custom node types. Supports animated edges for data flow visualization.
Backend aggregatorFastAPI + asyncioAsync WebSocket server that fans out to Kafka consumer, Redis, PostgreSQL, Neo4j simultaneously.
Kafka consumeraiokafkaAsync Kafka consumer that pushes events to WebSocket subscribers in real time.
StylingTailwind CSS + CSS variablesUtility classes for layout, CSS variables for consistent color theming (dark mode first).
LLM trace viewerCustom componentExpandable JSON viewer for prompt/response pairs. Syntax-highlighted. Diff view for consecutive decisions.
08 / Frontend Build Prompt
The Complete Prompt to Build the Dashboard

Use this prompt directly with Claude, GPT-4o, or any capable code generation LLM. It contains the full specification for the dashboard frontend. Pair it with this guide as a reference document.

SYSTEM CONTEXT

You are building the observability dashboard for a Cognitive Architecture system built through 11 phases: Perception Layer → Memory System (Working/Episodic/Semantic/Procedural) → Knowledge Graph → Temporal Engine → World Model → Reasoning Engine → Planning System → Execution Layer → Feedback Loop → Multi-Agent Coordination. The dashboard is the control surface that makes every internal state of this architecture visible in real time.

DESIGN AESTHETIC

Dark industrial-terminal aesthetic. Background: #0a0c0f. Primary accent: #00e5ff (cyan). Secondary: #7c3aed (purple). Success: #10b981. Warning: #f59e0b. Critical: #ef4444. Font stack: Space Mono for data/code, Syne or similar for labels. Minimal chrome. Maximum information density. Every element earns its pixel. No decorative gradients — function IS the design. Think: Bloomberg Terminal meets LCARS meets industrial SCADA. Panels have colored top-border accent strips per domain layer.

LAYOUT STRUCTURE

  • Top status bar (fixed, 48px): Architecture health badge (NOMINAL/DEGRADED/INCIDENT), event throughput sparkline (events/sec, last 60s), active agent count, reasoning queue depth, memory pressure %, Chaos Lab connection status, system uptime, current UTC time.
  • Left sidebar (200px, collapsible): Navigation to each panel section. Layer health dots (green/amber/red) next to each. Collapse to icon-only mode.
  • Main content area: Resizable panel grid. Default layout: 2-column on wide screens, single column on narrow. Each panel is independently scrollable with expand-to-fullscreen button.
  • Architecture flowchart (always-visible, collapsible): At the top of main area — the full data flow from signal source → perception → event bus → memory layers → world model → reasoning → planning → execution → feedback, rendered as an interactive React Flow diagram. Each node pulses/glows when actively processing. Click any node to jump to that panel.

PANEL 1 — KAFKA / EVENT BUS

  • Topic list: events, agent_events, actions, chaos_injections — each with messages/sec bar, consumer lag, partition count
  • Consumer group lag per group: memory-router, world-model, agent-monitor (show offset lag as number + color-coded bar)
  • Live event feed: scrolling list of last 50 events, each row shows: timestamp, source_id, event_type (color-coded by domain), severity badge, confidence score. Click to expand full JSON payload.
  • Dead-letter queue size + button to inspect DLQ contents
  • Replay controls: time range picker + "replay from here" button (calls backend endpoint)
  • Throughput chart: 5-minute sparkline of messages/sec across all topics, stacked by topic

PANEL 2 — PERCEPTION LAYER

  • Adapter status table: one row per adapter (log-agent, metrics-bridge, api-poller, sensor-mqtt, agent-event-sdk, chaos-webhook). Columns: adapter name, source count, events/min, validation pass%, last event timestamp, status (LIVE/STALE/ERROR).
  • Confidence score distribution: histogram of confidence values across events in last 5 minutes (bins: 0.0-0.5, 0.5-0.7, 0.7-0.9, 0.9-1.0)
  • Perception failures log: list of recent schema validation failures — event_id, source, failure reason, raw payload snippet
  • Entity extraction quality: entities found per event (avg), new entities discovered today, entity namespace size
  • Normalizer latency: p50/p95/p99 per adapter type, shown as horizontal bar chart

PANEL 3 — MEMORY SYSTEM (four sub-panels, tabbed)

Tab: Working Memory (Redis)

  • Utilization gauge (used MB / max MB), ops/sec (reads + writes), hit/miss ratio
  • Key space browser: sortable table of active keys — key name, size, TTL countdown (color red when <60s), type (hash/sorted_set/string)
  • Active goals list: each goal with status, created_at, priority
  • Current plan state: which plan is active, current step, steps remaining
  • Recent event window: last 15 events stored in working memory (name + timestamp + severity)

Tab: Episodic Memory (TimescaleDB + Qdrant)

  • Stats: total event count, events today, storage size (GB), ingest rate (events/min)
  • Query latency: list of last 10 queries with duration and type (time-range / similarity / pattern)
  • Event timeline: zoomable timeline showing event frequency over last 24h, colored by severity. Click a spike to query that time range.
  • Event type frequency chart: bar chart of most common event_types in last 24h
  • Similarity search results viewer: shows last Qdrant query (what the reasoning engine searched for) + top 3 results with similarity scores
  • Reasoning trace store: last 10 reasoning traces saved to episodic — expandable to see full prompt/response

Tab: Semantic Memory (Neo4j)

  • Graph stats: node counts by type (Service/Database/User/Agent/Task/Resource/Concept), edge counts by relationship type
  • Interactive graph: rendered with Sigma.js or D3 force layout. Node color by type. Edge thickness by weight. Zoom/pan/select. Click node for entity profile sidebar.
  • Entity profile sidebar: entity ID, type, properties table, connected entities list (with relationship type + weight), recent events referencing this entity
  • Recent graph mutations: list of last 20 node creates/updates and edge creates/updates with timestamp
  • Weak edges panel: edges with weight <0.2 (pruning candidates) shown separately
  • Cypher query runner: text input for Cypher queries, results displayed as table or sub-graph

Tab: Procedural Memory

  • Playbook library table: name, domain, success_rate (%), times_used, last_used, confidence_score. Sortable columns.
  • Top 5 active playbooks: larger cards showing name + most recent invocation context
  • Strategy confidence chart: bar chart of confidence scores for all playbooks, sorted descending
  • Recent reinforcements: log of strategies that were updated by the feedback loop this session

PANEL 4 — WORLD MODEL

  • Entity registry: searchable/filterable table — entity_id, type, health_status (badge), last_seen (time ago), trend arrow (↑↓→), confidence score. Highlight degraded/stressed rows.
  • Active anomalies list: severity-sorted cards — each shows entity, anomaly type, deviation magnitude, duration, and expandable causal chain (A→B→C arrows)
  • Open causal chains: visual representation of currently active causal chains — nodes (entities) connected by directed arrows, each arrow labeled with relationship type
  • Active event context: events still "open" (consequences unfolding) — event_id, event_type, affected entities, age
  • Uncertainty map: entities where confidence <0.7 highlighted. Shows what the world model is uncertain about.
  • World model update counter: total updates since start, updates in last minute, last update timestamp
  • Predictive state section: for each tracked metric — current value, predicted value at +5min / +15min / +60min, confidence of prediction, trend indicator

PANEL 5 — REASONING ENGINE

  • Last LLM call card: shows situation summary input (collapsible), full prompt (collapsible), full response (collapsible), parsed decision object as syntax-highlighted JSON, time-to-decision in ms, token counts (prompt + completion), estimated cost
  • Decision object fields prominently displayed: situation_assessment text, root_cause_hypothesis + confidence, recommended_action + parameters, requires_human_approval (red badge if true), alternative_actions list
  • Reasoning history: scrollable list of last 20 reasoning cycles — timestamp, situation summary (truncated), decision made, outcome (if known). Click to expand any.
  • Fast-path stats: rules fired vs LLM calls ratio (donut chart), list of fast-path rule matches with matched rule name
  • LLM performance: rolling avg latency (ms), cost accumulator ($), calls today, error rate

PANEL 6 — AGENT FLEET

  • Agent cards grid: one card per agent type (Monitor / Planner / Executor / Memory / Prediction / Specialist). Each card: - Agent name + icon + type badge - Status indicator: IDLE (gray) / ACTIVE (cyan pulse) / WAITING (amber) / ERROR (red) - Current task: short description or "—" if idle - Last action: name + result + timestamp - Stats: tasks completed today, avg task duration, error count - "View Trace" button → opens full trace log drawer
  • Conflict detection panel: if two agents have tasks targeting same entity — show conflict card with both agent names, conflicting plans, and resolution decision by planner
  • Agent spawn log: when and why specialist agents were spawned, their domain, lifetime, result
  • Communication log: inter-agent messages via event bus — from, to, event_type, timestamp

PANEL 7 — PLANNING & EXECUTION

  • Active plans list: each plan shows — goal text, status badge (RUNNING/PAUSED/COMPLETED/FAILED), progress bar (step N of M), current step name, time elapsed
  • Plan graph visualizer: DAG of the active plan's steps. Each step node: name, status color, duration. Edges show dependencies. Parallel branches shown side by side.
  • Approval gate queue: plans awaiting human approval. Each shows: plan goal, step requiring approval, risk assessment, recommended action, APPROVE and REJECT buttons. Approved/rejected decisions logged.
  • Execution action log: scrolling feed of actions executed — type (API/file/agent/message/human), target entity, params summary, status (SUCCESS/FAILED/RETRY), duration ms
  • Rollback history: plans that triggered rollback — what failed, what rollback steps ran, final state
  • Plan performance: success rate (last 100 plans), avg completion time by plan type, failure reason distribution

PANEL 8 — TEMPORAL STATE

  • Entity metric sparklines: for each tracked entity+metric pair — small chart showing last 30 data points with current value, trend direction (↑↓→), rate of change (%/min), deviation from baseline (sigmas). Color: green if normal, amber if drifting, red if anomalous.
  • Detected patterns list: each pattern card shows — type (SPIKE/DRIFT/CORRELATION/ABSENCE/RECURRENCE), entities involved, confidence score, detected_at, age, status (ACTIVE/RESOLVED)
  • Timeline builder: dropdown select entity + metric, date range picker, renders full time-series chart with baseline overlay, anomaly markers, and detected pattern annotations
  • Correlation matrix: heatmap of correlation coefficients between tracked metrics (last 24h). Click a cell to see the time-series overlay of both metrics.

PANEL 9 — PREDICTIONS

  • Active predictions table: metric_id, entity, horizon (5/15/60 min), predicted_value, confidence, model_type (Prophet/GNN/LLM), created_at
  • Prediction vs actual chart: for each completed prediction — scatter plot of predicted vs observed, with error bars. Hovering a point shows the context at prediction time.
  • Accuracy scorecard: MAE and MAPE per model type, per horizon. Rolling 7-day accuracy trend line.
  • GNN model status (if deployed): model version, training data size, last retrain date, inference latency, feature importance top-5
  • Upcoming predicted events: "In 12 minutes: auth-service p95 latency predicted to cross 500ms threshold (87% confidence)" — prioritized list

PANEL 10 — FEEDBACK LOOP

  • Recent feedback cycles: each cycle card — plan goal, action taken, expected outcome, observed outcome, match (SUCCESS/PARTIAL/FAILURE), world model delta summary, strategy update made
  • Strategy improvement chart: for each playbook type — rolling success rate over time (line chart, last 30 days). Should show upward trend as system learns.
  • Knowledge graph reinforcements: recent RESOLVES edges added or strengthened — "action:scale_replicas RESOLVES situation:db_pool_exhaustion (conf: 0.82)"
  • Compounding effect tracker: total feedback cycles completed, overall strategy improvement (first-week avg success rate vs last-week avg success rate), learning velocity
  • Observability traces: link to OpenTelemetry trace for each reasoning→execution→feedback cycle (opens Jaeger/Tempo in new tab)

PANEL 11 — CHAOS LAB

  • Target service configuration: URL input, connection test button, active adapter list showing which perception adapters are watching this target, connection health indicator
  • Fault injection controls: button grid for — latency_spike (slider: 100ms–10s), http_500_wave (slider: rate%), connection_timeout, memory_pressure, cpu_spike, queue_flood, dead_heartbeat, auth_failure_flood. Each fault has duration slider (10s–10min) and INJECT + STOP buttons.
  • Active faults display: currently injected faults as tags with remaining duration countdown and CANCEL button
  • Detection latency chain: for the most recent fault — timeline showing: fault_injected_at → first_kafka_event_at → world_model_anomaly_at → reasoning_invoked_at → plan_created_at → first_action_at. Shows each gap in milliseconds.
  • Full response trace: expandable step-by-step chronological log of the architecture's complete response to the fault — every event, memory write, world model update, reasoning decision, plan step, execution action, feedback event. Timestamps on each.
  • Scenario recorder: RECORD button to capture a complete fault→response scenario. REPLAY button to replay any saved scenario. Scenario library with saved replays.
  • Architecture grading: after each fault response — auto-grade: detection speed (A/B/C/F), correct root cause identification (Y/N), appropriate action selected (Y/N), execution success (Y/N), feedback loop closed (Y/N).

GLOBAL FEATURES (apply to all panels)

  • Every panel has: expand to fullscreen button, collapse to minimal/header-only mode, refresh rate selector (live / 1s / 5s / manual), and a "last updated" timestamp.
  • Global time range selector (top bar): affects all historical/query panels simultaneously (1h / 6h / 24h / 7d / custom)
  • Global entity search: type an entity ID anywhere in the top bar → all panels highlight that entity's data and the knowledge graph centers on it
  • Event inspector drawer: clicking any event_id anywhere in the dashboard opens a right-side drawer with the full event object, its trace through the system (which memory layers stored it, what world model updates it triggered, if it caused a reasoning invocation), and links to related events
  • Dark mode only. No light mode. Color accessibility: all status colors also use shape/icon differentiation, not color alone.
  • WebSocket reconnection: graceful handling of backend disconnection with reconnecting... state and auto-reconnect. No stale data shown — panels show "last known state + age" when disconnected.
  • Keyboard shortcuts: / to focus entity search, G to open graph full screen, A for agent fleet, C for chaos lab, Escape to close any drawer/fullscreen

BACKEND API CONTRACT

The frontend connects to a FastAPI backend at http://localhost:8000. WebSocket endpoint: ws://localhost:8000/ws/dashboard. REST endpoints for historical queries: /api/events?from=&to=&type=&entity= | /api/graph/query (POST, body: {cypher}) | /api/memory/working | /api/agents | /api/plans/active | /api/predictions | /api/feedback/recent | /api/chaos/scenarios. WebSocket pushes: {type: "event", data: UnifiedEvent} | {type: "world_model_update", data: WorldModelDelta} | {type: "agent_status", data: AgentStatus[]} | {type: "reasoning_result", data: DecisionObject} | {type: "plan_update", data: PlanState}

IMPLEMENTATION NOTES

  • Start with a mock data layer that generates realistic synthetic events so the frontend can be built and tested without the full backend running.
  • Use CSS Grid for panel layout with configurable column spans. Store layout preferences in localStorage.
  • All charts must handle empty/loading/error states gracefully.
  • The knowledge graph visualization is the most complex component — build it as a standalone component with its own data fetching. Use WebGL renderer (Sigma.js) for performance with 500+ nodes.

  • Approval gate panel is the ONLY panel with write operations — implement optimistic UI with rollback on error.
