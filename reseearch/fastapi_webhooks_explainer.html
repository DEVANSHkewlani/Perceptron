Why this design

The perception layer has two kinds of adapters. Polling adapters (API, Log, Database, File, Sensor) go out and fetch data on a schedule. Push adapters sit and wait for external systems to call them.

FastAPI is the push adapter. It is a web server that external systems POST data to. The moment something happens — a Prometheus alert fires, a user rage-clicks, a WAF blocks an IP — that system immediately sends an HTTP POST to your FastAPI server. No waiting for a poll cycle.

Think of it this way: polling adapters are like you checking your email every 30 seconds. The FastAPI server is like someone calling you directly the moment something happens. Push is always faster and cheaper than polling for real-time signals.

The FastAPI server is not a replacement for the other adapters. It handles only the sources that are designed to push — Alertmanager, Cloudflare, GitHub, your browser SDK. Everything else still uses its dedicated adapter.

Every single endpoint in main.py follows the exact same four-step lifecycle. Understanding this once means you understand all six endpoints.

Step 1 — receive and validate
@app.post("/perception/security-events", status_code=202) async def receive_security_events(request: Request, bg: BackgroundTasks): body = await request.json() # parse the raw POST body into a dict source = request.headers.get("X-Security-Source", "unknown") bg.add_task(_process_security_event, body, source) # hand off immediately return {"accepted": True} # return 202 RIGHT NOW — don't wait
FastAPI reads the body and immediately schedules the real work as a BackgroundTask. The calling system (Cloudflare, Alertmanager etc) gets its 202 response in milliseconds. It does not wait for Kafka, does not wait for the normalizer.
Step 2 — background task runs the normalizer
async def _process_security_event(raw: dict, source: str): source_id = f"security:{source}" # build namespaced source_id normalizer = SecurityEventNormalizer() # instantiate normalizer raw_copy = dict(raw) raw_copy.setdefault("source_system", source) result = normalizer.normalize(raw_copy, source_id) # pure function call await _publish(result) # send to Kafka
This runs after the 202 is already sent. The normalizer is a pure function — it takes a dict and returns either a CognitiveEvent or a PerceptionFailure. No network calls, no I/O inside the normalizer itself.
Step 3 — publish function routes to correct topic
async def _publish(event: CognitiveEvent | PerceptionFailure): payload = event.model_dump_json().encode("utf-8") # CognitiveEvent → cognitive.events # PerceptionFailure → cognitive.perception_failures topic = FAILURE_TOPIC if isinstance(event, PerceptionFailure) else KAFKA_TOPIC await producer.send_and_wait(topic, payload)
One function handles both outcomes. A valid event goes to the main topic. A failed normalization goes to the failures topic. Nothing is ever dropped.
Step 4 — Kafka holds it for downstream consumers

Once published to Kafka, the event is durable. The downstream reasoning engine, world model, and knowledge graph all consume from cognitive.events at their own pace. The perception layer's job is done.

AIOKafkaProducer — the Kafka connection
producer: AIOKafkaProducer | None = None @asynccontextmanager async def lifespan(app): global producer producer = AIOKafkaProducer(bootstrap_servers=KAFKA_BOOTSTRAP) await producer.start() # one connection, shared by ALL requests yield await producer.stop()

The producer is created once at startup and reused for every single request. This is critical — creating a new Kafka connection per request would add 200-500ms of overhead per event. A shared producer means publishing takes under 5ms. It's declared as a module-level global so all route handlers can access it.

If Kafka is not running when the server starts, producer stays None. The _publish function checks for this and logs a warning rather than crashing — the server still runs and normalizes, it just can't publish.
Lifespan context — startup and shutdown
@asynccontextmanager async def lifespan(app: FastAPI): # Everything BEFORE yield runs on startup producer = AIOKafkaProducer(...) await producer.start() yield # ← server runs here, handling requests # Everything AFTER yield runs on shutdown await producer.stop() # clean disconnect from Kafka app = FastAPI(lifespan=lifespan)

The lifespan pattern replaced the old @app.on_event("startup") in modern FastAPI. It guarantees the Kafka producer is cleanly disconnected when the server shuts down — no lost messages, no connection leaks. Without this, killing the server with Ctrl+C would leave a dangling TCP connection to Redpanda.

BackgroundTasks — why requests return immediately
@app.post("/perception/security-events", status_code=202) async def receive(request: Request, bg: BackgroundTasks): body = await request.json() bg.add_task(_process_security_event, body, source) # schedule return {"accepted": True} # return NOW — task runs after this

BackgroundTasks is a FastAPI feature that runs a function after the HTTP response is already sent. This matters because Cloudflare, Alertmanager, and GitHub all have short webhook timeouts (typically 5-10 seconds). If normalization + Kafka publishing took 3 seconds, you'd risk timeout errors from the caller. With BackgroundTasks, the caller gets 202 in under 10ms and the work happens asynchronously.

The background task runs in the same async event loop — it is not a thread or a separate process. It runs concurrently with other incoming requests, not after them.
Pydantic models — request validation
class UserEventBatch(BaseModel): events: list[dict[str, Any]] session_id: str user_id: str | None = None @app.post("/perception/user-events", status_code=202) async def receive_user_events(batch: UserEventBatch, bg: BackgroundTasks): ...

Only the user-events endpoint uses a typed Pydantic model for the request body. This means FastAPI automatically validates that events is a list and session_id is a string — and returns a 422 error automatically if not. The other endpoints use request.json() directly because their payload shapes are more variable (Alertmanager, GitHub webhooks all have complex nested schemas that are harder to model statically).

_publish — topic routing
KAFKA_TOPIC = "cognitive.events" FAILURE_TOPIC = "cognitive.perception_failures" async def _publish(event: CognitiveEvent | PerceptionFailure): if not producer: print(f"Producer offline. Dropped: {type(event)}") return payload = event.model_dump_json().encode("utf-8") topic = FAILURE_TOPIC if isinstance(event, PerceptionFailure) else KAFKA_TOPIC await producer.send_and_wait(topic, payload)

One shared function used by every endpoint. It makes two decisions: which topic to publish to (based on type), and what to do if Kafka is offline (log and drop rather than crash). send_and_wait means it waits for Kafka to acknowledge receipt before returning — this guarantees at-least-once delivery.

status_code=202 — why not 200

200 OK means "I processed your request and here is the result." 202 Accepted means "I received your request and I will process it, but I haven't yet." This is the correct HTTP semantic for async processing — the work happens in a BackgroundTask after the response is sent, so saying 200 would be a lie. Alertmanager and Cloudflare both understand 202 and treat it as success. Returning 200 would also work in practice, but 202 is more honest and helps with debugging — if you see a 200 from a webhook endpoint that uses BackgroundTasks, something is wrong with the design.

/health endpoint — self monitoring
@app.get("/health") async def health(): return { "status": "ok", "kafka": "connected" if producer and producer._sender else "disconnected", }

A lightweight endpoint that tells you whether the server is running AND whether its Kafka producer is connected. Your load balancer, Kubernetes liveness probe, or Prometheus scraper calls this. If Kafka goes down, {"kafka": "disconnected"} tells you events are being dropped without looking at logs.

Request headers — source identification
# Security events source = request.headers.get("X-Security-Source", "unknown") # → "cloudflare", "fail2ban", "custom-waf" etc # → becomes source_id: "security:cloudflare" # Deployment events event_type = request.headers.get("X-GitHub-Event", request.headers.get("X-GitLab-Event", "deployment")) # → "deployment", "push", "release" # → determines which event_type to produce

Custom HTTP headers carry metadata that isn't in the body. Cloudflare, GitHub, and GitLab all set standard headers on their webhooks. The server reads these to figure out which system sent the event and what kind it is — without needing each sender to include that in their JSON body. This is why you pass -H "X-Security-Source: cloudflare" in the curl commands.

The server has 6 push endpoints plus a health check. Each endpoint is a dedicated pipeline for one category of external system.

The difference between user-events and browser-events

These two are easy to confuse. user-events are intentional actions — clicks, form submits, checkout steps, search queries. They tell you what the user is trying to do. browser-events are environmental signals — JavaScript errors, Web Vitals scores, hydration failures. They tell you whether the browser environment is working correctly. A user can have a perfect session (all user-events look fine) while their browser is throwing hydration errors — these are separate concerns and separate normalizers.

Why FastAPI and not Flask, Django, or Express?

FastAPI is built on ASGI (Async Server Gateway Interface) which means it handles concurrent requests in a single thread using async/await — the same model as the rest of the perception layer (aiokafka, asyncpg, httpx are all async). Flask and Django are WSGI — they block per request and need threads or processes for concurrency. Since the perception layer is entirely async, FastAPI fits naturally. The Kafka producer is async, the normalizers are async-compatible, and BackgroundTasks is an async construct. Mixing Flask (sync) into an async codebase would create the same blocking problem we fixed in the queue adapter.

Why 202 and BackgroundTasks instead of just processing inline?

Alertmanager has a 10-second webhook timeout. If your normalization + Kafka publish takes longer than 10 seconds (e.g. Kafka is slow, normalizer hits a complex regex), Alertmanager marks the delivery as failed and retries — you get duplicate events. By returning 202 immediately and processing in the background, you decouple your processing time from the caller's timeout. The caller never knows or cares how long normalization takes.

Why a shared producer instead of creating one per request?

A Kafka producer connection involves a TCP handshake, metadata fetch, and authentication — roughly 200-500ms of setup time. If you created a new producer for every POST, a server handling 100 requests/second would spend most of its time connecting to Kafka rather than processing events. One shared producer, started at boot, makes every publish under 5ms.

Why does agent-events get confidence 1.0 while all other normalizers give lower scores?

Confidence represents how certain the perception layer is about its interpretation of the signal. Log lines are unstructured text matched with regex — lots of ambiguity, 0.75-0.85. Prometheus alerts are threshold crossings — deterministic math, 0.98. Agent events are produced by your own reasoning engine — it already knows exactly what it's reporting, there's no interpretation needed. Confidence 1.0 means "take this at face value, no uncertainty."

Why are there two separate endpoints for user and browser events instead of one?

They have different delivery patterns. User events are batched — the browser SDK collects 5 seconds of clicks and sends them together, so the endpoint receives a list. Browser events (JS errors, Web Vitals) are immediate — the moment an error fires or LCP is measured, it's sent instantly as a single event. Merging them into one endpoint would force you to handle both batch and single-event patterns in the same handler and the same normalizer, making both more complex for no benefit.
